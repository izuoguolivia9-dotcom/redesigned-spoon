# SOP: Updating Knowledge Base Articles
## Purpose
This SOP defines the standard process for reviewing, updating, and publishing knowledge base articles to ensure accuracy, consistency, and usability for end users.
## Scope
This SOP applies to all documentation updates made to user-facing knowledge base articles. It does not cover the creation of new documentation or major content restructures.

## Roles and Responsibilities
- **Technical Writer:** Reviews, updates, and formats knowledge base content according to documentation standards.
- **Subject Matter Expert (SME):** Validates technical accuracy and confirms that changes reflect current product behavior.
- **Documentation Manager (if applicable):** Reviews final updates and approves publication.

## Prerequisites
Before updating a knowledge base article, ensure:
- Access to the documentation platform or repository
- Familiarity with the applicable style guide
- Awareness of the target audience for the content

## Procedure

### Phase 1: Review Existing Content
1. Identify the article requiring an update.
2. Review the article for outdated, unclear, or incorrect information.
3. Note any gaps, inconsistencies, or user-facing issues.

### Phase 2: Gather and Validate Information
1. Consult relevant SMEs to confirm current product behavior or processes.
2. Review existing internal resources, such as product notes or support tickets.
3. Clarify assumptions and document any outstanding questions.

### Phase 3: Update the Article
1. Revise content to reflect validated information.
2. Use task-based language and consistent formatting.
3. Ensure terminology aligns with the approved style guide.
4. Add or update screenshots or examples if required.

### Phase 4: Review and Approval
1. Submit the updated article for SME review.
2. Address feedback and resolve any questions.
3. Obtain approval before publishing changes.

### Phase 5: Publish and Track Changes
1. Publish the updated article to the knowledge base.
2. Record the update date and summary of changes.
3. Monitor user feedback for potential follow-up updates.

All updates should be documented using version history or change logs to ensure traceability and accountability.
